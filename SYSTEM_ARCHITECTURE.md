# สถาปัตยกรรมระบบ CPR-Trainer Monitoring System (How It Works)

เอกสารฉบับนี้อธิบายถึงสถาปัตยกรรมของระบบ กลไกการทำงาน และเครื่องมือ (Tech Stack) ที่ใช้ในการพัฒนาระบบ CPR-Trainer Monitoring System อย่างละเอียด หลังจากผ่านการปรับโครงสร้าง (Refactoring) ให้เป็นไปตามหลักการ Software Engineering สมัยใหม่

---

## ภาพรวมของระบบ (System Overview)
ระบบนี้ถูกออกแบบมาเพื่อใช้ในการเรียนการสอนและประเมินผลการทำ CPR แบบเรียลไทม์ โดยระบบจะอ่านค่าเซ็นเซอร์จากหุ่นจำลอง ส่งมาประมวลผลที่เซิร์ฟเวอร์ แสดงกราฟบนหน้าเว็บแบบสดๆ และบันทึกข้อมูลเพื่อใช้ประเมินผลและออกรายงาน (Excel/CSV) ได้ในภายหลัง

ระบบแบ่งออกเป็น 4 ส่วนหลัก (4-Tier Architecture):
1. **Hardware (ESP32):** ตัวรับรู้และวัดค่าจากเซ็นเซอร์
2. **Backend (Node.js):** ตัวกลางประมวลผล สื่อสาร และกระจายข้อมูล
3. **Database (PostgreSQL):** แหล่งเก็บข้อมูลถาวร
4. **Frontend (Web/HTML5):** ส่วนแสดงผลสำหรับผู้ใช้งาน

---

## 1. Hardware Component (ฝั่งฮาร์ดแวร์)
**เครื่องมือที่ใช้:** ESP32 Microcontroller, เซ็นเซอร์ FSR (วัดแรงกด) และเซ็นเซอร์ระยะทาง (วัดความลึก)

**กลไกการทำงาน:**
* ESP32 จะคอยอ่านค่าแรงกดและความลึกในการกดหน้าอกของผู้ฝึก
* นำข้อมูลมาคำนวณอัตราการกด (BPM - Beats Per Minute) และประเมินคุณภาพเบื้องต้น (เช่น "GOOD RATE", "PUSH HARDER")
* จากนั้น ESP32 จะแพ็กข้อมูลทั้งหมดให้อยู่ในรูปแบบ **JSON** แล้วส่งผ่านสาย USB (Serial Port) มายังคอมพิวเตอร์อย่างต่อเนื่องในระดับมิลลิวินาที

---

## 2. Backend Component (ฝั่งเซิร์ฟเวอร์)
เซิร์ฟเวอร์คือหัวใจหลักในการจัดการข้อมูลที่ได้รับมาจาก ESP32 เพื่อกระจายไปยังหน้าเว็บและจัดเก็บลงฐานข้อมูล 
**เครื่องมือที่ใช้:**
* **Node.js:** รันไทม์หลักของระบบ
* **Express.js:** Web Framework สำหรับสร้าง RESTful API
* **SerialPort (`serialport`):** ไลบรารีสำหรับคุยกับ ESP32 ผ่าน COM Port
* **WebSocket (`ws`):** สำหรับการส่งข้อมูลแบบ Real-time ไปที่หน้าจอเว็บ
* **PostgreSQL (`pg`):** ไดรเวอร์เชื่อมต่อฐานข้อมูล
* **ExcelJS (`exceljs`):** เครื่องมือสร้างไฟล์รายงาน .xlsx
* **Jest:** เครื่องมือเขียนเทสต์สำหรับทำ TDD

**กลไกการทำงาน (อิงตามสถาปัตยกรรมใหม่):**
* **`services/serial.service.js` (ผู้ฟังฮาร์ดแวร์):**
  คอยดักจับข้อมูล JSON จาก COM Port ทันทีที่ข้อมูลมาถึง มันจะประมวลผล จัดเก็บลงฐานข้อมูล (ถ้ากำลังมีการบันทึก Session) และกระจาย Event ต่อไป
* **`services/state.service.js` (หน่วยความจำชั่วคราว):**
  เก็บข้อมูลล่าสุด (Latest Data) และแบตเตอรี่ไว้ใน Memory (RAM) หากหน้าเว็บทำการ Refresh ระบบสามารถดึงข้อมูลชุดนี้ไปแสดงผลได้ทันทีโดยไม่ต้องร้องขอจาก Database
* **`services/websocket.service.js` (สถานีถ่ายทอดสด):**
  รอรับ Event จาก Serial Service แล้ว Broadcast (กระจายสัญญาณ) ข้อมูลแบบสดๆ ไปยังทุกเบราว์เซอร์ที่เปิดหน้า Dashboard อยู่
* **`routes/*` (ประตูเชื่อมต่อ API):**
  แยกแยะเส้นทางต่างๆ เช่น `/api/auth` (ล็อกอิน), `/api/cpr` (ขอข้อมูลสถิติ), `/api/sessions` (จัดการรอบการฝึกและ Export ไฟล์)
* **`utils/excel.util.js` (คนงานสร้างไฟล์):**
  รับข้อมูลดิบจากฐานข้อมูล นำมาวาดตาราง และแปลงเป็นไฟล์ Excel (.xlsx) อย่างสวยงาม

---

## 3. Database Component (ฝั่งฐานข้อมูล)
**เครื่องมือที่ใช้:** PostgreSQL

**กลไกการทำงาน:**
ฐานข้อมูลถูกออกแบบมาเป็นแบบ Relational Database (RDBMS) เก็บข้อมูลเป็น 3 ตารางหลัก:
1. **`users`:** เก็บข้อมูล Admin สำหรับล็อกอินเข้าระบบ (ใช้ `bcrypt` เข้ารหัสรหัสผ่าน)
2. **`sessions`:** เก็บข้อมูลรอบการฝึกแต่ละครั้ง (ชื่อผู้ฝึก, เวลาเริ่ม-จบ, สถิติรวมและคะแนนโดยรวม)
3. **`compressions`:** เก็บข้อมูลการกดหน้าอก **ทุกๆ ครั้ง** (ความลึก, แรง, อัตราเร็ว) ซึ่งผูกกับ ID ของตาราง sessions 

*เมื่ออาจารย์กดปุ่ม "จบการฝึก" (End Session) ระบบ Backend จะสั่งให้ PostgreSQL ทำการรวบรวม (Aggregate) ข้อมูล `compressions` นับร้อยๆ รายการ เพื่อหาค่าเฉลี่ยและตัดเกรดออกมาบันทึกลงตาราง `sessions` ในพริบตา*

---

## 4. Frontend Component (ฝั่งผู้ใช้งาน)
**เครื่องมือที่ใช้:** HTML5, CSS3, Vanilla JavaScript, Chart.js (สำหรับวาดกราฟ)

**กลไกการทำงาน:**
* ทันทีที่เข้าสู่หน้า Dashboard โค้ด JavaScript ฝั่ง Client จะทำการสถาปนาการเชื่อมต่อ **WebSocket** ไปยังเซิร์ฟเวอร์
* ทุกๆ วินาทีที่มีการกดหน้าอก ข้อมูลจะถูกยิงมาที่เบราว์เซอร์ผ่าน WebSocket โดยอัตโนมัติ (ไม่ต้อง Refresh)
* ฟังก์ชันในไฟล์ `app_v2.js` จะรับข้อมูลเหล่านั้นไปแยกย้ายตามจุดต่างๆ เช่น:
  * นำค่า `force` และ `depth` ไปวาดจุดต่อจุดบนกราฟของ **Chart.js** ให้วิ่งไปเรื่อยๆ (Live Chart)
  * นำข้อความสถานะไปอัปเดตบนหน้าจอ (เช่น "GOOD" เป็นสีเขียว, "PUSH HARDER" เป็นสีแดง)
* หน้าเว็บยังมีหน้าที่สื่อสารกับ REST API เพื่อสั่งเริ่มบันทึกคะแนน (Start Session) และขอออกรายงาน (Export Batch Excel) อีกด้วย

---

## สรุป (Conclusion)
ระบบ CPR-Trainer Monitoring System ทำงานประสานกันแบบกึ่ง Asynchronous โดยใช้ **Event-Driven Architecture** ตรงกลาง (ผ่าน Event Emitter และ WebSocket) ทำให้ระบบสามารถรับส่งข้อมูลความถี่สูงจาก Hardware ไปสู่หน้าจอ UI ได้อย่างลื่นไหล ไม่มีอาการค้าง (Non-blocking) และมีการจัดเรียงโค้ดแบบซ่อนความซับซ้อน (Encapsulation) ช่วยให้ดูแลรักษาง่ายต่อยอดได้ในอนาคต
